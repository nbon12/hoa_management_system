# Contract: Client Type Generation & Drift Gate

**Applies to**: the frontend/backend API contract (FR-011, FR-012). Types-only generation per Clarifications (2026-07-02).

## Source of truth

The backend's OpenAPI document, generated by the existing FastEndpoints.Swagger (NSwag) pipeline from the implemented endpoints.

## Export mechanism

`dotnet run --project HOAManagementCompany -- --export-openapi <output-path>` — a startup flag (same pattern as the existing `--seed`) that writes the OpenAPI JSON and exits without binding the HTTP server or requiring a database.

## Generation

- Tool: `openapi-typescript` (dev-dependency in `neko-hoa`).
- Output: `neko-hoa/src/app/core/api/generated-types.ts` — types only (zero runtime code). Committed to the repository.
- Entry point: `npm run generate:api-types` (runs export + codegen).

## Consumption rules

- Hand-written services in `core/services/` keep their structure but type requests/responses with generated types (directly or via thin re-exports from `core/models`).
- `core/models/index.ts` holds only app-internal view-models; it MUST NOT redefine shapes that exist in the generated file.
- Storybook fixtures and specs type their stubs with the generated/canonical types (FR-012) — no hand-written parallel shapes.

## Drift gate (CI, PR verification pipeline)

1. Build backend → run the export flag → run `openapi-typescript`.
2. `git diff --exit-code -- neko-hoa/src/app/core/api/generated-types.ts`.
3. Non-empty diff fails verification with instructions to run `npm run generate:api-types` and commit.

Failure modes: unavailable/invalid OpenAPI export fails the gate loudly (no fallback to the stale committed file — spec Edge Cases).

- SC-004: deliberate server-side shape change without regeneration → gate fails; duplicate/dead client definitions = 0.
